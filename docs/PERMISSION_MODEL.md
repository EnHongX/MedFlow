# 权限模型

## 该不该做

该做，但 MVP 只做基础 RBAC。

## 为什么

系统有患者、医生、前台、管理员 4 类用户。权限边界如果不提前定清楚，后续接口和页面很容易混乱。

## 怎么做

## 角色

| 角色 | 说明 |
| --- | --- |
| `PATIENT` | 患者 |
| `DOCTOR` | 医生 |
| `FRONTDESK` | 前台/导诊台 |
| `ADMIN` | 管理员 |

## 权限范围

| 角色 | 权限范围 |
| --- | --- |
| 患者 | 只能查看和操作自己的预约、排队进度、就诊记录 |
| 医生 | 只能查看自己排班下的预约、患者摘要、接诊记录 |
| 前台 | 处理当日到诊、分诊、叫号、过号、转科 |
| 管理员 | 维护科室、医生、排班、号源、规则 |

## NestJS 实现建议

- 使用 `AuthGuard` 校验登录。
- 使用 `RolesGuard` 校验角色。
- 使用装饰器声明接口角色，例如 `@Roles('ADMIN')`。
- 患者、医生的数据范围不能只靠角色判断，还要校验资源归属。

## MVP 不做

- 不做细粒度按钮权限。
- 不做动态权限菜单。
- 不做多租户权限。
- 不做复杂组织架构权限。
