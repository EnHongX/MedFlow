# 技术栈选择

## 该不该做

该定，但只定项目启动需要的主技术栈，不做复杂基础设施设计。

## 为什么

技术栈需要支撑 4 个端共用一套业务 API，重点是类型清晰、开发效率高、数据库模型可维护。

## 怎么做

| 层 | 选择 | 说明 |
| --- | --- | --- |
| 后端运行时 | `Node.js` | 生态成熟，适合中后台 API |
| 后端语言 | `TypeScript` | 类型约束，降低多人协作成本 |
| 后端框架 | `NestJS` | 模块化清晰，适合业务 API |
| ORM | `Prisma` | Schema 清晰，迁移和类型生成方便 |
| 数据库 | `PostgreSQL` | 关系模型强，适合预约、排班、队列等业务 |
| 前端框架 | `Vue 3` | 适合后台和业务系统 |
| 构建工具 | `Vite` | 本地开发快 |
| 前端语言 | `TypeScript` | 与后端保持类型风格一致 |
| UI 组件 | `Element Plus` | 管理后台和业务端开发效率高 |

## 不做

- 不拆微服务。
- 不引入消息队列。
- 不引入 Kubernetes。
- 不提前做数据仓库。
- 不做复杂权限平台。
