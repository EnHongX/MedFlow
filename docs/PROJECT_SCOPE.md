# 项目范围

## 该不该做

该做，但第一阶段只做门诊预约与分诊协同的最小闭环，不做完整医院信息系统。

## 为什么

单纯挂号系统价值有限，门诊现场真正的协同问题发生在预约、到诊、分诊、接诊、规则维护之间。

第一阶段如果把范围压到“患者预约 -> 前台签到分诊 -> 医生接诊 -> 管理员维护基础规则”，可以验证核心业务价值，避免一开始陷入完整 HIS、EMR、支付、医保等重型系统。

按当前需求，最容易不合理的地方不是业务方向，而是过早做服务端口和微服务拆分。MVP 应先保证 4 个端的主流程跑通。

## 怎么做

第一阶段只覆盖以下闭环：

```text
患者预约
-> 系统锁定号源/时段
-> 前台确认到诊
-> 前台分诊/叫号
-> 医生接诊并更新状态
-> 管理员维护科室、医生、排班、号源规则
```

## MVP 必做

- 患者端：登录、预约、取消、改约、查看状态和排队进度
- 医生端：查看预约、接诊、查看病人记录
- 前台端：到诊、叫号、分诊、过号处理
- 管理后台：科室、排班、号源、规则维护
- 患者创建、取消、改期预约
- 前台签到、分诊、叫号、过号
- 医生查看队列、更新接诊状态、填写病历摘要

## MVP 可延后

- 外部短信、微信、App Push 等真实通知渠道
- 复杂统计面板
- 复杂权限矩阵
- 多院区、多租户
- 完整审计后台

## 暂不做

- 在线支付
- 医保对接
- 电子处方
- 检验检查申请
- 完整电子病历
- 多院区复杂调度
- 智能推荐医生
- 患者端复杂社区/内容运营

## 风险

- ⚠️ 时间黑洞：过早做完整医疗业务系统。
- ⚠️ 技术过度设计：未验证流程前拆太细、引入复杂中间件。
- ⚠️ 收益不匹配：先做低频管理功能，忽略预约和分诊主链路。
- ⚠️ 伪需求：把“服务端口、微服务边界”当作第一阶段必须完成的设计。
